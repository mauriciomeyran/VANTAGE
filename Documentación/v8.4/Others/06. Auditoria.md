## FASE 1 — Comprensión del Sistema

### Flujo principal punta a punta

**Capa L1/L2 (Active/Strategic, semanal):**
1. Humano descubre vacantes → entrega JSON con campos Class A
2. AI Component deduplica (30d window) + normaliza → genera DRY RUN
3. Humano emite `APROBAR_WRITE` → AI escribe Class A en Notion
4. Python corre `vantage_pipeline.sh` → calcula Score, Gate, campos Class B
5. Human revisa Ready-to-Apply → decide postulación

**Capa L3 (Passive, continuo):**
1. `mail_pipeline.py` IMAP-fetch Gmail → Groq extrae Class A → escribe Notion directamente
2. Python corre en ciclo → calcula Class B
3. Human revisa (sin gate AI-side)

**Runtime (L0, on-demand):**
1. Operador invoca `vantage.py ask/context/resolve/query/sync`
2. Query Layer busca en entity_index_v2.json (snapshot)
3. Resolver confirma existencia en Notion vivo
4. Context Layer trae propiedades + bloques
5. Agent API interpreta intención y enruta a handler

### Invariantes del sistema — ¿realmente invariantes?

| Invariante (Kernel §1) | ¿Es real? | Excepción no documentada |
|------------------------|-----------|--------------------------|
| "Sin URL válida no entra al pipeline" | Parcial | Bypass (Inbound/Referencia/Networking) lo omite — está documentado, pero el contrato de qué valida el operador antes de emitir Bypass no está especificado. |
| "Score lo calcula Python, no AI" | Sólido | Sin excepción detectada en código. |
| "Gate no se sobreescribe manualmente" | Parcial | RT-1 corrige inputs Class A para que Python *recalcule* — lo cual produce un gate diferente. En la práctica es un bypass de facto del gate original; la documentación lo llama "recuperación", no "sobreescritura", pero el resultado funcional es equivalente. |
| "Strategy es responsabilidad humana" | Sólido | Sin excepción. |

### Componentes: manual vs automatizable

| Componente | Estado |
|------------|--------|
| Ingesta L1/L2 — discovery de vacantes | **100% manual** (humano) |
| Entrega JSON → DRY RUN → APROBAR_WRITE | **Manual** (humano aprueba) |
| Escritura Class A post-aprobación | Automatizable / ya automatizado |
| Cálculo Class B (score, gate) | **Automático** (Python) |
| Ingesta L3 (mail) | **Automático** (mail_pipeline.py) |
| Regeneración entity_index | Manual (o `sync` desde 2026-06-16) |
| Decisión de postulación | **Siempre manual** |
| Revisión BLOCKED / RT-1 | **Manual** |

### Fallo de ingesta pasiva L3

**El sistema no detecta el fallo de L3.** Si `mail_pipeline.py` cae silenciosamente (fallo de conexión IMAP, Groq API down, expiración de credenciales), no hay mecanismo de alerta documentado en ninguno de los cinco scripts auditados. El pipeline continúa ciegamente procesando solo L1/L2 sin notificar la brecha. Esta es la ausencia más seria de monitoreo del sistema.

### Supuesto de memoria del AI Component

El sistema asume **memoria cero entre sesiones**. Cada sesión recibe Kernel + SP como contexto frío. No hay mecanismo para persistir el estado de una sesión DRY RUN parcialmente completada. Si el usuario cierra Claude a mitad de un DRY RUN, la sesión se pierde — no hay protocolo de reanudación documentado.

### Mecanismos de recuperación: ¿legítimos o workarounds?

**RT-1** (recuperación BLOCKED) es un mecanismo legítimo con un caso de uso real: vacantes que fallaron el gate por problemas de datos (campo Class A corrupto, URL mal formada) que pueden ser corregidas. No es un bypass sistemático. La distinción es válida.

**`vacante_purge_trash_only.py`** (ya deprecado per Changelog §8.3) *sí era* un workaround para un bug de atomicidad en `vacante_purge.py`. El bug fue resuelto. La deprecación es correcta.

## FASE 2 — Auditoría de Riesgos

### A) Riesgos de Integridad de Datos

**[RID-01] · Integridad · Crítico**
Descripción: `entity_index_v2.json` es un snapshot estático. Si `vantage.py sync` no se ejecuta después de que `feed_processor.py` escribe nuevas entradas en Notion, el index está stale. `query`/`search_entities`/`find_candidates` operan sobre datos desactualizados sin ninguna advertencia al usuario.
Impacto: Roles nuevos no aparecen en `find_candidates`; roles archivados siguen apareciendo como activos.
Probabilidad: **Alta** — el ciclo semanal L1/L2 agrega entradas regularmente.

**[RID-02] · Integridad · Alto**
Descripción: El Resolver (`resolver_layer_v1.py`) ejecuta `POST /v1/data_sources/{id}/query` **sin filtro de page_id**. Trae *todos* los resultados de la DB y luego busca el `page_id` en la lista. Si la DB tiene >100 páginas y el endpoint no pagina automáticamente, el `page_id` podría no aparecer en los primeros results y el resolver devolvería `not_found` para una entidad que sí existe.
Impacto: `context` y `resolve` fallan en entidades válidas. Pipeline de CV-A trabajaría con datos incompletos.
Probabilidad: **Alta** a 384+ entidades en Archivo Tracker (ya confirmado en index).

**[RID-03] · Integridad · Alto**
Descripción: Dedup cross-layer entre L3 y L1/L2 depende de que `feed_processor.py` aplique la jerarquía secuencialmente. Si L3 procesa una entrada antes de que el array L1/L2 del lunes sea entregado, puede crear un duplicado en Notion que la dedup del lunes no elimina (porque la ventana de 30d usada por AI Component opera sobre el JSON entrante, no sobre el estado actual de Notion).
Impacto: Duplicados en el Tracker que Python procesará dos veces, produciendo dos entidades con el mismo hash en entity_index si `sync` se corre entre los dos writes.
Probabilidad: **Media**.

**[RID-04] · Integridad · Medio**
Descripción: `APROBAR_WRITE` autoriza escritura de campos Class A únicamente (Kernel §4). Sin embargo, el contrato no especifica qué sucede si el DRY RUN presentado al humano contiene un campo Class B que se coló en la normalización. El token de aprobación no discrimina: autoriza todo el payload presentado.
Impacto: Un campo Class B en el DRY RUN puede ser escrito en Notion si el operador aprueba sin revisar exhaustivamente.
Probabilidad: **Baja** (requiere bug en normalización), pero consecuencia alta.

### B) Riesgos de Comportamiento del AI Component

**[RAI-01] · Comportamiento AI · Crítico**
Descripción: Regla de Oro #4 dice "no delegar escritura al usuario". Las excepciones son PDF y Google Drive. Sin embargo, si la Notion API falla post-APROBAR_WRITE, el sistema no tiene protocolo documentado. Un AI Component que siga la letra de la regla podría reintentar indefinidamente o simplemente reportar el error y cerrar sesión — pero el operador no sabe qué quedó escrito y qué no. No hay registro de transacción parcial.
Impacto: Estado inconsistente en Notion después de una falla mid-write.
Probabilidad: **Media** (fallas de red son comunes).

**[RAI-02] · Comportamiento AI · Alto**
Descripción: Las Reglas de Oro no cubren el caso edge de **un prompt que mezcla un trigger válido con una solicitud prohibida**. Ejemplo: "CV-A esta vacante y además dime si vale la pena aplicar". El sistema debe rechazar la segunda parte (Regla #1) pero ejecutar la primera. No hay instrucción explícita sobre comportamiento de split en este caso. Un AI puede ejecutar ambas, ejecutar solo la primera en silencio, o rechazar todo.
Impacto: Comportamiento inconsistente entre sesiones, especialmente bajo modelos distintos.
Probabilidad: **Media**.

**[RAI-03] · Comportamiento AI · Alto**
Descripción: El gate DRY RUN → aprobación explícita es robusto en teoría, pero el contrato de "tokens de aprobación válidos" incluye `Ok` y `Go` — respuestas que pueden aparecer naturalmente en conversación ("Ok, entendido" o "Go ahead with the next step"). No hay mecanismo para distinguir un `Ok` como aprobación de escritura de un `Ok` conversacional.
Impacto: Escritura no intencionada en Notion si el contexto conversacional produce un token de aprobación ambiguo.
Probabilidad: **Media** — especialmente en sesiones largas donde el DRY RUN no está inmediatamente antes del Ok.

**[RAI-04] · Comportamiento AI · Alto**
Descripción: Durante CV-A, el AI Component debe cruzar keywords del JD contra el Career Canon y reportar `fit_gaps`. La Regla #4 de no inferir datos faltantes se aplica, pero si el JD tiene información incompleta sobre el rol (descripción vaga, sin requisitos técnicos explícitos), el AI puede reportar `fit_gaps` vacíos o erróneos sin señalarlo como incertidumbre. La Regla de Oro protege contra *inventar* datos pero no contra *omitir señalar ambigüedad*.
Impacto: CVs generados con `fit_gaps` incorrectos que no reflejan la realidad del JD.
Probabilidad: **Media**.

**[RAI-05] · Comportamiento AI · Medio**
Descripción: Los campos prohibidos para AI (`Score`, `Gate_Decision`, etc.) están documentados en Kernel §4. Sin embargo, el contrato dice "si el JSON entrante incluye campos Class B, se **ignoran**" — no dice "se detectan y se alerta". Un AI Component puede procesar silenciosamente payloads contaminados sin notificar al operador que hubo contaminación.
Impacto: El operador no sabe que sus feeds externos están enviando campos Class B que el sistema descarta.
Probabilidad: **Media** (fuentes externas vía L2 pueden incluir campos de clasificación propios).

### C) Riesgos Operativos

**[ROP-01] · Operativo · Crítico**
Descripción: No existe protocolo documentado para fallo de `mail_pipeline.py` (L3). No hay health check, no hay alerta, no hay log de último run exitoso accesible al operador. La ingesta pasiva puede estar caída días o semanas sin que el operador lo note.
Impacto: Vacantes de alta señal enviadas por email (Inbound/Referencia) no ingresan al sistema. El operador decide sin visibilidad de esa información.
Probabilidad: **Alta** (credenciales IMAP/Groq expiran, servicios caen).

**[ROP-02] · Operativo · Alto**
Descripción: El sistema no tiene campo de "fecha de último sync" en ningún lugar accesible al operador durante operación normal. El operador no sabe cuántos días tiene de stale su entity_index sin correr `vantage.py status` y revisar `metrics`.
Impacto: Decisiones de CV-A y `find_candidates` basadas en datos desactualizados sin advertencia.
Probabilidad: **Alta**.

**[ROP-03] · Operativo · Alto**
Descripción: Los Hard Blocks (SP §6: L'Oréal, Levi's, Palacio de Hierro) están declarados en el SP para Claude pero **no están implementados en ningún script Python**. `feed_processor.py` y `mail_pipeline.py` pueden escribir entradas de empresas bloqueadas en Notion. La exclusión solo opera si Claude está en el loop — L3 (automático, sin Claude) no tiene este filtro.
Impacto: Entradas de empresas hard-blocked pueden aparecer en el Tracker vía L3, contaminando las vistas y el conteo de roles activos.
Probabilidad: **Alta** — L3 es continuo y automático.

**[ROP-04] · Operativo · Medio**
Descripción: `_handle_show_roles` y `_handle_find_candidates` en `agent_api.py` iteran sobre todas las entidades de `VANTAGE_TRACKER` haciendo `assemble_context()` por cada una — lo que implica Resolver call + page fetch + blocks fetch por entidad. Con 81 entidades y throttle de 0.35s/request, estas operaciones toman entre 30–90 segundos. A 40+ vacantes activas, la latencia aumenta proporcionalmente sin ningún mecanismo de paginación o early-exit.
Impacto: `find_candidates` y `show_roles` se vuelven inutilizables en pipelines con alta carga activa. No hay timeout configurado.
Probabilidad: **Media-Alta** con el crecimiento natural del Tracker.

**[ROP-05] · Operativo · Medio**
Descripción: Los mecanismos de exclusión (Hard Blocks) no tienen enforcement en el Entity Index. Una entidad de L'Oréal que ya está en Notion aparecerá en los resultados de `search_entities` y `find_candidates` a menos que el AI Component la filtre por nombre en cada query — lo cual no está especificado como comportamiento.
Impacto: El operador recibe resultados de empresas excluidas en sus consultas Runtime, produciendo confusión o trabajo manual de filtrado.
Probabilidad: **Media**.

## FASE 3 — Fricciones de Diseño

### Fricciones de activación
El operador tiene que saber qué comando usar de `vantage.py` (`ask`, `query`, `context`, `resolve`, `sync`). El Manual §1.8 documenta los prompts reconocidos de `ask`, pero no hay un árbol de decisión: "¿tengo el entity_id? → usa `context`. ¿solo texto? → usa `ask`". La tabla §1.8 requiere que el operador conozca las keywords exactas que disparan cada intent — hay casos edge no documentados (ej: ¿"muéstrame mis candidatos" dispara `find_candidates`?).

### Fricciones de mantenimiento
Tres acciones requieren disciplina constante o el sistema se degrada:
- `sync` después de cada ciclo L1/L2 (entity_index se vuelve stale)
- Revisar y resolver entradas `REVIEW_NEEDED` en Notion (no hay recordatorio automático)
- Actualizar Career Canon con `CANON-UPDATE` después de cada CV generado (si no se hace, el próximo CV-A puede generar gaps incorrectos)

### Fricciones de recuperación (pausa de 2 semanas)
Si el operador no usa el sistema por 2 semanas acumulan:
- Entity index stale (nuevas entradas L3 no en el index)
- Posibles entradas REVIEW_NEEDED sin resolver
- `resolver_registry_v2.json` potencialmente desactualizado si cambia estructura de Notion
- `.bak` files acumulados en `Layer_1/scripts/`
No hay procedimiento de "arranque frío" documentado que guíe la reactivación sistemática.

### Fricciones de escala (10 → 40 vacantes)
- `_handle_find_candidates`: O(n) llamadas a Notion. Lineal con entidades activas.
- Entity Index en memoria: no hay problema hasta ~10k entidades, pero la falta de TTL hace que el cache nunca expire.
- Las vistas filtradas de Notion (el Tracker como UI) escalan bien — el bottleneck es siempre el lado Python/Runtime.

### Fricciones de contexto
El operador necesita tener en la cabeza simultáneamente:
- En qué capa están sus vacantes activas (L1/L2/L3, con su estado en Notion)
- Si el entity_index está fresco
- Qué triggers están activos en Claude vs Python
- El estado del Career Canon (qué versión fue usada para el último CV)

Con 30+ vacantes y múltiples CVs en progreso, esta carga cognitiva es sustancial para un solo operador.

### Fricciones de handoff AI↔pipeline
El flujo `DRY RUN → APROBAR_WRITE → write` es claro en teoría. La fricción existe post-aprobación: si el pipeline falla silenciosamente (Notion API timeout, red caída), el operador no recibe feedback hasta que investiga. No hay confirmación de escritura exitosa en el formato de sesión actual.

### Fricciones de recuperación BLOCKED
El flujo RT-1 está documentado en Kernel §5.4 pero no hay un prompt o comando específico que lo active. El operador debe saber que debe: (1) editar el campo en Notion, (2) cambiar Status a Target, (3) correr `vantage_pipeline.sh`. Este flujo de tres pasos manuales sin checklist guiado es una fricción real bajo presión.

## FASE 4 — Mejoras Sustanciales

**[M-01] — Staleness indicator en `vantage.py status`**
Qué cambia: `status()` incluye `index_age_hours` calculado desde el timestamp de modificación de `entity_index_v2.json`. Si `index_age_hours > 24`, el output incluye `"warning": "entity_index_stale"`.
Por qué importa: Resuelve RID-01 y ROP-02 — el operador sabe cuándo necesita `sync`.
Implementación: `os.path.getmtime(ENTITY_INDEX_PATH)` en `vantage.py::status()`. Una línea.
Trade-off: Ninguno significativo.
Prioridad: **Inmediata**.

**[M-02] — Paginación en `_query_notion` del Resolver**
Qué cambia: `resolver_layer_v1.py::_query_notion` itera sobre todos los resultados paginados de `data_sources/{id}/query` antes de buscar el `page_id`. Actualmente trae un solo batch.
Por qué importa: Resuelve RID-02. Con 384+ entidades en Archivo Tracker, el resolver falla en entidades que no están en la primera página de resultados.
Implementación: Añadir loop while `has_more` con `start_cursor` en `_query_notion`, igual al patrón ya implementado en `agent_api.py::_notion_db_query`.
Trade-off: Latencia adicional proporcional al tamaño de la DB en resoluciones individuales. Mitigable con caché de resultados por data_source_id dentro de la sesión.
Prioridad: **Inmediata**.

**[M-03] — Hard Blocks en feed_processor.py**
Qué cambia: `feed_processor.py` añade un check de `HARD_BLOCKED_BRANDS` antes de escribir en Notion. Lista incluye "L'Oréal", "Levi's", "El Palacio de Hierro" (y variantes case-insensitive).
Por qué importa: Resuelve ROP-03. Sin esto, L3 ingesta vacantes de empresas bloqueadas sin filtro.
Implementación: Dict/set `HARD_BLOCKED_BRANDS` al tope del módulo. Guard en `process_record()` antes de la escritura.
Trade-off: Requiere mantener la lista sincronizada con SP §6 HARD BLOCKS. Riesgo de divergencia si SP cambia sin actualizar el código.
Prioridad: **Inmediata**.

**[M-04] — Token de aprobación contextual (evitar ambigüedad `Ok`/`Go`)**
Qué cambia: Remover `Ok` y `Go` de la lista de tokens válidos de `APROBAR_WRITE`. Mantener solo `APROBAR_WRITE`, `APROBAR`, `SÍ`, `sí`, `YEP`, `yep`. O alternativamente, requerir que el token aparezca como respuesta directa a la pregunta "¿Confirmar escritura? Escribe APROBAR_WRITE".
Por qué importa: Resuelve RAI-03. `Ok` y `Go` son demasiado comunes en conversación para ser tokens de aprobación seguros.
Implementación: Editar lista de tokens en Kernel §4 y SP §3. No requiere cambio de código Python (el gate está en el AI Component).
Trade-off: Ligera fricción adicional para el operador en aprobaciones rápidas.
Prioridad: **Inmediata**.

**[M-05] — Health check de L3 (mail_pipeline.py)**
Qué cambia: `mail_pipeline.py` escribe un archivo `l3_heartbeat.json` con timestamp de último run exitoso. `vantage.py status` lo lee y reporta `l3_last_success` y `l3_status: ok/stale/unknown`.
Por qué importa: Resuelve ROP-01 — el fallo silencioso de L3 se vuelve visible.
Implementación: Una línea de escritura JSON en `mail_pipeline.py` post-run. Una línea de lectura en `vantage.py::status()`.
Trade-off: Requiere que `mail_pipeline.py` esté en el mismo host que `vantage.py` o que el heartbeat sea accesible vía path compartido.
Prioridad: **Próximo ciclo**.

**[M-06] — `find_candidates` con in-memory pre-filter**
Qué cambia: `_handle_find_candidates` en `agent_api.py` aplica primero `list_entities(source_db="VANTAGE_TRACKER")` con filtros de status/gate/match sobre el entity_index (in-memory, sin tocar Notion), y solo llama `assemble_context()` sobre el subset filtrado.
Por qué importa: Resuelve ROP-04. De O(n) llamadas Notion a O(k) donde k << n.
Implementación: Añadir campos `status`, `gate_decision`, `match` al entity_index durante `sync` (ya están disponibles en Notion). Esto requiere actualizar `generate_entity_index_v2.py` para incluir esos campos.
Trade-off: Entity index se vuelve más pesado. Tradeoff controlable: estos son campos de texto cortos.
Prioridad: **Próximo ciclo**.

**[M-07] — Alertar (no ignorar silenciosamente) campos Class B en payload entrante**
Qué cambia: Cuando AI Component detecta campos Class B en el JSON entrante durante normalización, agrega al DRY RUN un bloque `"class_b_detected": ["score", "gate_decision"]` como advertencia visible al operador.
Por qué importa: Resuelve RAI-05. El operador sabe que sus fuentes están enviando datos que el sistema descarta.
Implementación: Añadir Regla de Oro auxiliar: "Si campos Class B detectados en input, reportar en DRY RUN como advertencia sin bloquear el flujo."
Trade-off: Puede generar ruido en feeds limpios. Mitigable limitando la advertencia a la primera ocurrencia por fuente.
Prioridad: **Próximo ciclo**.

**[M-08] — Procedimiento de arranque frío documentado**
Qué cambia: Añadir sección "Reactivación tras pausa" en Manual §7 (Troubleshooting) con checklist: (1) `vantage.py status` → revisar staleness. (2) `vantage.py sync`. (3) Revisar REVIEW_NEEDED en Notion. (4) Confirmar L3 heartbeat. (5) Verificar `resolver_registry_v2.json` vigente.
Por qué importa: Reduce la fricción de recuperación para un operador solo.
Implementación: Edición de documentación pura.
Trade-off: Ninguno.
Prioridad: **Próximo ciclo**.

## FASE 5 — Veredicto Ejecutivo

### Fortalezas del sistema

1. **Separación de ownership Class A/B es arquitectónicamente sólida.** El contrato de que Python es el único que toca campos Class B elimina una clase entera de errores de data corruption por componente AI. Es una decisión de diseño correcta y bien implementada.
2. **El protocolo DRY RUN → APROBAR_WRITE es el mecanismo de control más importante del sistema y está bien definido.** Produce un checkpoint humano real antes de cualquier escritura, sin fricción excesiva.
3. **La cadena de resolución entity_index → Resolver → Context es elegante y correctamente estratificada.** La separación entre snapshot rápido (Query Layer) y verificación en vivo (Resolver) es una decisión de ingeniería inteligente que balancea performance e integridad.

### Vulnerabilidad crítica principal

**El Resolver no pagina `data_sources/query` — falla silenciosamente en Archivo Tracker (384 entidades) devolviendo `not_found` para entidades válidas.** [RID-02]

### La mejora de mayor impacto

**Añadir campos de status/gate/match al entity_index durante `sync` [M-06]** — elimina el bottleneck O(n) de Notion calls en `find_candidates` y transforma Runtime de herramienta lenta a herramienta de sub-segundo para las queries más frecuentes.

### Puntaje de solidez: 6.5/10

El core del sistema (pipeline de scoring, separación de ownership, protocolo de escritura) está bien diseñado. Sin embargo, los gaps de documentación desincronizada, la ausencia de monitoreo de L3, el bug del resolver en DBs grandes, y la dependencia cognitiva del operador para mantener el index fresco son fricciones reales que acumulan deuda operativa bajo uso sostenido.

### ¿Tiene vida útil real? **Condicionado**

Puede sostener una búsqueda de 3–6 meses si: (1) se implementan M-01 (staleness indicator) e M-02 (paginación del resolver) en la próxima sesión, (2) se establece disciplina de `sync` post-ciclo semanal, y (3) los patches del Roadmap Fase 1 se insertan efectivamente en los documentos canónicos. Sin esas condiciones, la deuda documental y el bug del resolver producirán fallos progresivos que degradarán la confianza del operador en los datos.

### Veredicto sobre mecanismos de recuperación

**RT-1 es un componente legítimo**, no un bypass sistemático. Existe para un caso de uso real y bien acotado (corrección de Class A corrupta), opera dentro del pipeline (requiere que Python recalcule), y no viola ningún invariante. Lo que sí es sintomático es que la *ausencia de un protocolo de arranque frío documentado* hace que el operador tienda a usar RT-1 como herramienta de diagnóstico general, no solo para BLOCKED — y ahí empieza a comportarse como workaround. La solución es M-08, no cambiar RT-1.
