### 7.2. RUNTIME SCOPE — INTEGRACIÓN CON TRIGGERS

#### 7.2.1 Qué Existe

VANTAGE Runtime es una capa de **lectura/consulta** adicional al sistema CV/Tracker. Vive en `Layer_1/scripts/` y **no participa** en los triggers `[CV-B, CANON-UPDATE]` ni en FEED.

#### 7.2.2 Cuándo Invocarla

VANTAGE Runtime se invoca cuando el operador pide explícitamente una consulta de datos que el **Entity Index** o **Notion** puede resolver sin pasar por el flujo CV ni por SYNC. Ejemplos:

- "¿Cuántos roles activos hay?" → `python3 vantage.py ask "show active roles"`.
- "Tráeme el contexto completo de TRACKER:H_xxxx" → `python3 vantage.py context TRACKER:H_xxxx`.
- "Compara estas dos vacantes" → `python3 vantage.py ask "compare TRACKER:H_x TRACKER:H_y"`.

**Diferencia clave con SYNC:**

- SYNC entrega **máximo 12 líneas tabulares sin interpretación** (Regla de Oro 4).
- Runtime puede devolver **objetos completos** (`context`) o **agregados** (`ask "find candidates"`).

#### 7.2.3 Qué Esperar de Runtime

- **Output:** Siempre en JSON estructurado (nunca prosa libre).
- **Operaciones:** Solo lectura — **nunca escribe en Notion**.
- **Latencia:** Puede tardar más que SYNC en operaciones masivas (ej: `ask "show active roles"` itera 81 entidades con throttle de 0.35s/request).
- **Errores:** Explicitos (`ResolverError` con `status`/`error`), nunca silenciosos ni inferidos.

#### 7.2.4 Cómo Razonar Usando Entidades

Un `entity_id` (`TRACKER:H_xxxx`, `ARCHIVO:H_xxxx`, etc.) es la **unidad atómica de referencia**. Flujo correcto:

1. `query "<texto>"` o `ask "<texto>"` (fallback search) → Obtener `entity_id` candidato(s) por coincidencia de slug en `page_url`.
2. Si hay un **match claro** → `context <entity_id>` para detalle completo, o `resolve <entity_id>` si solo se necesita confirmar existencia/URL.
3. Si hay **múltiples matches** → Presentar las opciones (`page_url` de cada una) antes de elegir.

#### 7.2.5 Cuándo Usar Lookup vs. Search vs. Context Retrieval

| Necesidad | Función | Toca Notion | Ejemplo |
| --- | --- | --- | --- |
| Tengo el `entity_id`/`canonical_id` exacto | `find_entity` | No | `query "TRACKER:H_93a9bae7f01e656e"` |
| Tengo texto parcial (rol, marca) | `search_entities` | No | `ask "VM Coordinator Gucci"` |
| Necesito confirmar entidad viva + URL | `resolve` | Sí (1 query) | `resolve TRACKER:H_93a9bae7f01e656e` |
| Necesito propiedades + contenido | `context` | Sí | `context TRACKER:H_93a9bae7f01e656e` |
| Necesito agregados/listas | `ask "<prompt>"` | Sí | `ask "show active roles"` |

#### 7.2.6 Integración con Triggers Existentes

**Trigger: QA [PDF]**

- **Con Runtime:**
    1. Extraer entidades del PDF (ej: `Rol`, `Marca`).
    2. Invocar Runtime: `vantage_ask(f"resolve role:{Rol} brand:{Marca}")`.
    3. Usar `runtime_data` para validar el HANDOFF.
- **Código de ejemplo:**
    
    ```python
    def qa_trigger(pdf_path):
        entities = extract_entities_from_pdf(pdf_path)
        runtime_data = vantage_ask(f"resolve role:{entities['Rol']} brand:{entities['Marca']}")
        handoff = generate_handoff(pdf_path, runtime_data)
        return handoff
    ```
    

**Trigger: SYNC**

- **Con Runtime:** Reemplazar llamadas directas a Notion por `vantage_ask("list active roles")`.

**Triggers donde NO usar Runtime:** `[CV-B, CANON-UPDATE]`.
