# ADR-001: Runtime Graph Builder

## Status
Accepted

## Context
A forensic investigation (2026-07-04) revealed that VANTAGE Runtime contains a historical technical debt: graph artifacts (`graph_v2.json`, `backlinks_v2.json`) were manually introduced during Runtime initialization and never became part of the automated Runtime Build process.

### Current State
- `entity_index_v2.json` is regenerated by `python3 vantage.py sync`
- `graph_v2.json` and `backlinks_v2.json` are consumed by `graph_layer.py` and `agent_api.py`
- No Graph Builder has ever existed in the repository
- Git history confirms no deleted/renamed generator script
- Graph files appeared fully-formed in commit 4a67bd0 (2026-06-15)

### Architectural Violation
The Runtime violates its own architecture:
- `vantage.py sync()` regenerates only `entity_index_v2.json`
- Graph artifacts are derived from entity index but not regenerated
- This creates staleness risk and inconsistency

## Decision
Implement the first official Runtime Graph Builder by extending `generate_entity_index_v2.py` to generate graph artifacts as part of the entity index generation process.

### Why This Approach
1. **Architectural alignment**: Graph edges are derived from entity index metadata (hash-based relationships between ARCHIVO and TRACKER entities)
2. **Single source of truth**: Graph generation belongs with its data source
3. **Minimal change**: Extends existing script rather than adding new component
4. **Solves inconsistency**: Eliminates staleness risk by regenerating with source
5. **Maintains contract**: `sync()` already exists and is the expected regeneration mechanism

### Why Not Other Approaches
- **Restore Graph Builder**: No evidence original builder existed; would require discovering/inventing generation logic
- **Generate on demand**: Doesn't solve staleness problem; adds latency; violates read-only caching pattern
- **New Runtime layer**: Violates constraint to not redesign Runtime

## Consequences
### Positive
- Runtime Build becomes complete and deterministic
- All three artifacts (`entity_index_v2.json`, `graph_v2.json`, `backlinks_v2.json`) belong to same Runtime Build
- Eliminates technical debt of manual graph maintenance
- Documentation becomes accurate (files are now actually "generados por script")
- Atomic write pattern ensures no partial updates

### Negative
- Increases `generate_entity_index_v2.py` complexity
- Ties graph generation to entity index regeneration frequency
- Adds validation overhead to sync operation

### Neutral
- No changes to public contracts
- No changes to JSON schemas
- No changes to Runtime layers (Resolver, Query, Context, Agent API)
- Graph Layer remains read-only consumer

## Implementation
The Graph Builder shall:
1. Load `entity_index_v2.json` after entity generation
2. Build `graph_v2.json` with `archived_from` relationships only
3. Build `backlinks_v2.json` as inverse representation of graph edges
4. Use only deterministic evidence from entity metadata (hash matching)
5. Validate: no orphan entity_ids, every edge references existing nodes, backlinks exactly match graph
6. Use atomic temporary files + replace() for failure recovery

## Execution Order
```
1. Generate entity_index_v2.json
2. Generate graph_v2.json
3. Generate backlinks_v2.json
4. Validate artifacts
5. Reload Runtime caches
6. status()
```

## Validation Criteria
- No orphan entity_ids in graph edges
- Every edge references existing nodes in entity index
- Backlinks exactly match graph (inverse relationship)
- Graph generation is deterministic (same input → same output)
- Atomicity: failure preserves previous artifacts
