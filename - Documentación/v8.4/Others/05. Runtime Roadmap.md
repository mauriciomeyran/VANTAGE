### 2. PROPÓSITO DEL SISTEMA

El sistema VANTAGE está diseñado para optimizar la gestión de vacantes en el sector de Luxury Retail mediante la automatización de procesos de análisis, clasificación y priorización. El núcleo del sistema se basa en la interacción entre tres capas principales: **L1 — Active Recon**, **L2 — Strategic Search** y **L3 — Passive Intake**, con un flujo de datos jerárquico que garantiza la deduplicación y consistencia de la información.

VANTAGE Runtime es una capa adicional de **lectura/consulta** que opera sobre las bases de Notion (VANTAGE Tracker, Archivo Tracker, Archivo Dry Run, Bug Tracker) para reducir la latencia y el consumo de tokens en operaciones de consulta. ## 12. RUNTIME COMPONENTS — ESPECIFICACIÓN TÉCNICA

### 12.1 Integración con la Arquitectura Existente

VANTAGE Runtime se integra como una capa de consulta **read-only** sobre el sistema CV/Tracker. No altera los triggers existentes (`[CV-A, CV-B, QA, FAST, CANON-UPDATE, SYNC, STATUS]`), pero extiende las capacidades de resolución de entidades para reducir tokens y latencia.

### 12.2 Flujo de Resolución de Entidades

1. **Input:** `entity_id` (ej: `TRACKER:H_93a9bae7f01e656e`) o `canonical_id` (hash SHA-256).
2. **Lookup en `entity_index_v2.json`:** Obtiene `source_db` (ej: `VANTAGE_TRACKER`) y `page_id`.
3. **Mapeo a `data_source_id`:** Usa `resolver_registry_v2.json` para mapear `source_db` a `data_source_id` (ej: `VANTAGE_TRACKER` → `442938be-fc42-828f-b72e-076818d65a5b`).
4. **Query a Notion:** Ejecuta `POST /v1/data_sources/{data_source_id}/query` para confirmar que `page_id` existe.
5. **Output:**
    
    ```json
    {
      "entity_id": "TRACKER:H_93a9bae7f01e656e",
      "status": "resolved",
      "source_db": "VANTAGE_TRACKER",
      "page_url": "https://app.notion.com/p/...",
      "resolved": true
    }
    ```
    

### 12.3 Contratos de Datos

- **`entity_id`:** Formato `PREFIX:H_<hash16>` (ej: `TRACKER:H_93a9bae7f01e656e`).
- **`canonical_id`:** Hash SHA-256 completo (64 caracteres hex).
- **`source_db`:** Identificador de la base de origen (ej: `VANTAGE_TRACKER`, `ARCHIVO_TRACKER`).
- **`data_source_id`:** UUID de Notion (ej: `442938be-fc42-828f-b72e-076818d65a5b`).

### 12.4 Responsabilidades y Límites

- **Resolver Layer:** Solo resuelve entidades que existen en el Entity Index. **No descubre entidades nuevas.**
- **Query Layer:** Opera 100% sobre el snapshot en memoria (`entity_index_v2.json`).
- **Context Layer:** Combina snapshot (para `page_id`) + Notion en vivo (para propiedades/contenido actuales).
- **Agent API:** Interpreta intenciones en lenguaje natural.

### 12.5 Integración con el Kernel Existente

- **No modifica:** Triggers `[CV-A, CV-B, QA, FAST, CANON-UPDATE, SYNC, STATUS]` (gestionados por `feed_processor.py`).
- **Extiende:** Capacidad de consulta **read-only** sobre Notion.
- **Boundary:** Runtime **no escribe** en Notion (solo `feed_processor.py` lo hace).

## 12. RUNTIME SCOPE — INTEGRACIÓN CON TRIGGERS

### 12.1 Qué Existe

VANTAGE Runtime es una capa de **lectura/consulta** adicional al sistema CV/Tracker. Vive en `Layer_1/scripts/` y **no participa** en los triggers `[CV-B, CANON-UPDATE]` ni en FEED.

### 12.2 Cuándo Invocarla

VANTAGE Runtime se invoca cuando el operador pide explícitamente una consulta de datos que el **Entity Index** o **Notion** puede resolver sin pasar por el flujo CV ni por SYNC. Ejemplos:

- "¿Cuántos roles activos hay?" → `python3 vantage.py ask "show active roles"`.
- "Tráeme el contexto completo de TRACKER:H_xxxx" → `python3 vantage.py context TRACKER:H_xxxx`.
- "Compara estas dos vacantes" → `python3 vantage.py ask "compare TRACKER:H_x TRACKER:H_y"`.

**Diferencia clave con SYNC:**

- SYNC entrega **máximo 12 líneas tabulares sin interpretación** (Regla de Oro 4).
- Runtime puede devolver **objetos completos** (`context`) o **agregados** (`ask "find candidates"`).

### 12.3 Qué Esperar de Runtime

- **Output:** Siempre en JSON estructurado (nunca prosa libre).
- **Operaciones:** Solo lectura — **nunca escribe en Notion**.
- **Latencia:** Puede tardar más que SYNC en operaciones masivas (ej: `ask "show active roles"` itera 81 entidades con throttle de 0.35s/request).
- **Errores:** Explicitos (`ResolverError` con `status`/`error`), nunca silenciosos ni inferidos.

### 12.4 Cómo Razonar Usando Entidades

Un `entity_id` (`TRACKER:H_xxxx`, `ARCHIVO:H_xxxx`, etc.) es la **unidad atómica de referencia**. Flujo correcto:

1. `query "<texto>"` o `ask "<texto>"` (fallback search) → Obtener `entity_id` candidato(s) por coincidencia de slug en `page_url`.
2. Si hay un **match claro** → `context <entity_id>` para detalle completo, o `resolve <entity_id>` si solo se necesita confirmar existencia/URL.
3. Si hay **múltiples matches** → Presentar las opciones (`page_url` de cada una) antes de elegir.

### 12.5 Cuándo Usar Lookup vs. Search vs. Context Retrieval

| Necesidad | Función | Toca Notion | Ejemplo |
| --- | --- | --- | --- |
| Tengo el `entity_id`/`canonical_id` exacto | `find_entity` | No | `query "TRACKER:H_93a9bae7f01e656e"` |
| Tengo texto parcial (rol, marca) | `search_entities` | No | `ask "VM Coordinator Gucci"` |
| Necesito confirmar que la entidad sigue viva + URL | `resolve` | Sí (1 query) | `resolve TRACKER:H_93a9bae7f01e656e` |
| Necesito propiedades + contenido completo | `context` | Sí | `context TRACKER:H_93a9bae7f01e656e` |
| Necesito agregados/listas interpretadas | `ask "<prompt>"` | Sí | `ask "show active roles"` |

    3. Usar `runtime_data` para validar el HANDOFF.

| Error | Causa | Solución |
| --- | --- | --- |

### 7.7 FAQ

**¿VANTAGE Runtime puede escribir en Notion?**

No. Todas las operaciones son GET. La escritura sigue siendo territorio exclusivo de `feed_processor.py` bajo el protocolo **DRY RUN → APROBAR_WRITE**.

**¿Necesito regenerar el Entity Index antes de cada consulta?**

No. El Entity Index (`entity_index_v2.json`) es un snapshot estático cacheado en memoria. Se regenera manualmente con `generate_entity_index_v2.py`.

**¿Qué pasa si una entidad fue modificada en Notion después de generar el index?**

- `resolve` y `context` consultan Notion en vivo para esa entidad puntual (reflejan el estado actual).
- `query`/`search` operan sobre el snapshot (entidades nuevas/eliminadas no aparecerán hasta regenerar el index).

### 7.8 Advertencias

1. **Runtime NO reemplaza a `feed_processor.py`:**
    - `feed_processor.py` sigue siendo el único que escribe en Notion (Class A/Class B).
    - Runtime es **solo lectura**.
2. **Dependencia del Entity Index:**
    - Si `entity_index_v2.json` está desactualizado, Runtime devolverá datos obsoletos.
    - **Solución:** Ejecutar `python3 vantage.py sync`.
3. **Campos Class B:**
    - Runtime **lee** `Gate_Decision`, `Score`, etc., pero **nunca los modifica**.
    - Claude debe ignorarlos en el HANDOFF (ya están resueltos).
